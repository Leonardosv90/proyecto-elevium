# Vision de Plataforma Elevium

Ultima actualizacion: 30 de mayo de 2026

Este documento resume las ideas planteadas para evolucionar Elevium desde un sitio web comercial hacia una plataforma digital mas completa, funcional y escalable.

## Como usar este documento

Este documento no es una lista cerrada de tareas. Es una guia viva para tomar decisiones de producto y desarrollo sin perder la vision general.

La recomendacion es revisarlo por etapas:

1. Elegir una fase.
2. Convertirla en tareas pequenas.
3. Desarrollar, probar y publicar.
4. Medir resultados.
5. Ajustar antes de pasar a la siguiente fase.

La idea central es construir despacio, pero con direccion.

## Vision general

Elevium debe crecer como un ecosistema que combine:

- Estrategia digital.
- Marketing y contenido.
- Marca personal y autoridad profesional.
- Inteligencia artificial y automatizacion.
- Produccion audiovisual.
- Produccion fisica de marca: merch, impresion, sublimacion, estampado y material publicitario.
- Recursos, capacitaciones y membresias.

La meta no es tener solo una pagina bonita, sino construir un sistema comercial y educativo que pueda captar prospectos, mostrar autoridad, administrar contenido, vender productos/servicios y formar una comunidad.

## Lineas principales del ecosistema

### 1. Website comercial

El sitio actual funciona como base publica de Elevium.

Debe evolucionar para incluir:

- Home clara y orientada a conversion.
- Metodo ELEVAR interactivo.
- Servicios con informacion ampliada.
- Casos de estudio detallados.
- Diagnostico estrategico.
- Recursos descargables.
- Contacto y WhatsApp.
- Seccion de produccion de marca / merch corporativo.

Rol del sitio:

- Explicar rapidamente que hace Elevium.
- Generar confianza.
- Mostrar metodo, servicios y casos.
- Capturar prospectos.
- Llevar a diagnostico, WhatsApp o contacto.
- Servir como base para futuras areas privadas.

### 2. Metodo ELEVAR interactivo

La pagina del metodo puede volverse mas funcional.

Idea planteada:

- Cada letra del metodo ELEVAR debe ser clickeable.
- Al hacer click, se abre una ventana flotante/modal.
- El modal muestra el desarrollo completo de esa fase.
- Debe tener navegacion de siguiente/anterior.
- Si es la primera fase, no se muestra anterior.
- Si es la ultima fase, no se muestra siguiente.

Fases:

- E: Explorar.
- L: Leer.
- E: Estructurar.
- V: Visibilizar.
- A: Acelerar.
- R: Refinar.

Contenido posible dentro de cada fase:

- Objetivo de la fase.
- Que se analiza.
- Que entregables produce.
- Ejemplos.
- Imagenes o visuales de apoyo.
- CTA hacia diagnostico o contacto.

### 3. Servicios con detalle ampliado

La pagina de servicios debe dejar de ser solo un catalogo breve.

Idea planteada:

- Al dar click en un servicio, se abre una ventana o pagina de detalle.
- Cada servicio debe explicar como funciona.
- Puede incluir imagenes, ejemplos, procesos, entregables y resultados esperados.

Servicios principales:

- Marca personal y autoridad digital.
- Estrategia y gestion de contenido.
- Branding y diseno digital.
- Produccion audiovisual.
- Web y captacion digital.
- IA y automatizacion.
- Capacitacion y mentoria.
- Produccion de marca / merch corporativo.

Cada detalle de servicio podria tener:

- Para quien es.
- Problemas que resuelve.
- Que incluye.
- Como es el proceso.
- Entregables.
- Ejemplos visuales.
- Preguntas frecuentes.
- CTA contextual.

### 4. Casos de estudio dinamicos

La pagina de casos debe evolucionar hacia un sistema actualizable.

Idea planteada:

- Agregar casos periodicamente.
- Al hacer click en un caso, mostrar el desarrollo completo.
- Incluir fotos, imagenes, videos, texto, resultados y proceso.

Cada caso podria tener:

- Nombre del proyecto.
- Cliente o tipo de cliente.
- Problema inicial.
- Objetivo.
- Proceso realizado.
- Servicios aplicados.
- Galeria de imagenes.
- Videos.
- Antes y despues.
- Resultados.
- Aprendizajes.
- CTA para solicitar un diagnostico similar.

Necesidades administrativas:

- Crear casos.
- Editar casos.
- Ocultar casos.
- Publicar casos.
- Eliminar casos.
- Marcar casos como destacados.
- Subir imagenes y videos.
- Ordenar casos.

### 5. Panel administrativo

Para que Elevium sea mantenible, se propuso crear un panel privado.

Funciones del panel:

- Gestionar casos.
- Gestionar servicios.
- Gestionar recursos.
- Gestionar leads.
- Ver diagnosticos enviados.
- Revisar metricas.
- Publicar/ocultar contenido.
- Administrar cursos en el futuro.

Usuarios del panel:

- Administrador principal.
- Equipo interno futuro.
- Posibles colaboradores de contenido o soporte.

Estados utiles:

- Borrador.
- Publicado.
- Oculto.
- Destacado.
- Archivado.

Permisos futuros:

- Admin: puede ver y modificar todo.
- Editor: puede crear y editar contenido, pero no tocar configuraciones criticas.
- Soporte/comercial: puede ver leads, diagnosticos y solicitudes.

### 6. Diagnostico estrategico

El diagnostico debe ser practico y no abrumador.

Principios definidos:

- Que se sienta rapido.
- Que tenga pocas acciones.
- Que no parezca un formulario interminable.
- Que primero entregue valor y luego pida datos.
- Que mida autoridad digital con respuestas simples.

Flujo recomendado:

1. Preguntas rapidas.
2. Resultado.
3. Datos minimos de contacto.

Datos recomendados:

- Nombre.
- WhatsApp.
- Correo.
- Marca o negocio.
- Servicio de interes opcional.
- Prioridad principal.

Evitar pedir demasiados campos al inicio para no perder usuarios en el embudo.

### 7. Recursos y lead magnets

La seccion de recursos debe convertirse en una biblioteca actualizable.

Ideas planteadas:

- Guias.
- Herramientas.
- Agentes IA.
- Tutoriales.
- Cursos gratis.
- Checklists.
- Plantillas.
- Recursos descargables.
- Mini capacitaciones.

Objetivo:

- Captar leads.
- Educar al usuario.
- Posicionar a Elevium como autoridad.
- Alimentar el embudo hacia diagnostico, servicios o membership.

Cada recurso podria tener:

- Titulo.
- Descripcion.
- Categoria.
- Archivo descargable.
- Imagen destacada.
- Formulario de captura.
- Tags.
- Estado publicado/oculto.

### 8. Analitica y registro de visitas

Se planteo la idea de registrar informacion relevante de visitantes para tomar mejores decisiones.

Datos utiles:

- Pagina visitada.
- Fecha y hora.
- Origen de trafico.
- UTM source, medium, campaign.
- Dispositivo.
- Navegador.
- Sistema operativo.
- Pais/ciudad aproximada.
- Clicks importantes.
- Descargas.
- Formularios enviados.
- Clics a WhatsApp.
- Ruta de navegacion.

Consideraciones importantes:

- La IP y datos del dispositivo pueden considerarse datos personales.
- Se debe crear politica de privacidad.
- Puede requerir aviso de cookies o consentimiento segun el uso.
- Conviene capturar solo datos utiles, no todo por curiosidad.

Eventos prioritarios para medir primero:

- Visita a pagina principal.
- Click en solicitar diagnostico.
- Inicio del diagnostico.
- Resultado generado.
- Diagnostico enviado.
- Click a WhatsApp.
- Descarga de recurso.
- Click en servicio.
- Click en caso.

La analitica debe ayudar a responder preguntas concretas:

- Que pagina convierte mejor.
- Que servicio genera mas interes.
- Donde se abandona el diagnostico.
- Que recursos capturan mas leads.
- Desde que fuente llegan los mejores prospectos.

Opciones:

- Usar herramientas como Vercel Analytics, Plausible o PostHog.
- Crear analitica propia con Supabase.
- Combinar eventos propios con una herramienta externa.

Crear una plataforma de membresia donde usuarios puedan acceder a:

- Seminarios.
- Modulos.
- Lecciones.
- Recursos exclusivos.
- Progreso.
- Certificados.
- Gamificacion.

Funciones posibles:

- Registro/login de usuarios.
- Cursos gratis y pagos.
- Modulos por curso.
- Lecciones por modulo.
- Videos.
- Archivos descargables.
- Progreso por usuario.
- Insignias.
- Puntos.
- Certificados.
- Comentarios o comunidad.
- Acceso segun plan.

Esto convertiria Elevium en una plataforma educativa, no solo en una agencia/firma de servicios.

Posicionamiento posible:

- Merch corporativo.
- Productos publicitarios.
- Material de marca.
- Produccion fisica para empresas.
- Kits corporativos.
- Branding fisico.

Nombre posible de la linea:

- Elevium Brand Lab.
- Elevium Merch.
- Elevium Print.
- Elevium Produccion.
- Elevium Corporate Merch.

Oferta recomendada:

- Kits de bienvenida para empresas.
- Merch para eventos.

- Senaletica.

## Prioridades recomendadas

### Prioridad 1 - Conversion del sitio actual

Antes de construir una plataforma grande, el sitio debe convertir mejor.

Tareas:

- Mejorar interacciones del Metodo ELEVAR.
- Ampliar detalle de servicios.
- Mejorar casos.
- Terminar el diagnostico como embudo simple.
- Conectar formularios a una herramienta real.
- Medir clicks principales.

### Prioridad 2 - Contenido administrable

Cuando el sitio ya tenga una base comercial clara, conviene hacer editable lo que mas cambiara.

Tareas:

- Casos dinamicos.
- Recursos dinamicos.
- Servicios dinamicos.
- Panel admin basico.
- Subida de imagenes.

### Prioridad 3 - Datos y seguimiento comercial

Despues de captar prospectos, hay que poder darles seguimiento.

Tareas:

- Base de leads.
- Diagnosticos guardados.
- Estados comerciales.
- Notas internas.
- Origen de cada lead.
- Dashboard simple.

### Prioridad 4 - Produccion de marca B2B

Integrar el negocio de productos personalizados como una linea estrategica de Elevium.

Tareas:

- Pagina de produccion de marca.
- Catalogo inicial.
- Paquetes B2B.
- Solicitud de cotizacion.
- Formulario para subir logo/arte.

### Prioridad 5 - Membership

La membresia es una vision poderosa, pero conviene desarrollarla cuando ya exista audiencia, recursos y demanda.

Tareas:

- Login.
- Cursos.
- Modulos.
- Lecciones.
- Progreso.
- Certificados.
- Gamificacion.
- Pagos.

## Recomendacion tecnologica

Por la vision planteada, el proyecto puede evolucionar hacia:

```text
Next.js + Supabase + Vercel
```

### Por que Next.js

- Permite construir sitio publico y plataforma privada.
- Buen soporte para SEO.
- Rutas dinamicas.
- Componentes reutilizables.
- APIs internas.
- Integracion natural con Vercel.
- Escala mejor hacia dashboard, membership y contenido dinamico.

### Por que Supabase

- Base de datos PostgreSQL.
- Autenticacion de usuarios.
- Storage para imagenes, archivos y videos.
- Reglas de seguridad.
- Panel de datos.
- Buena base para leads, casos, recursos, cursos y analitica.

### Por que Vercel

- Deploy automatico desde GitHub.
- Ideal para Next.js.
- Dominios faciles de configurar.
- Preview deployments.
- Buen flujo de desarrollo.

## Ruta de desarrollo recomendada

### Fase 1 - Mejorar el sitio actual

Objetivo:

Validar experiencia e interacciones sin migrar todo de golpe.

Acciones:

- Modal interactivo del Metodo ELEVAR.
- Detalles ampliados de servicios.
- Mejorar casos como experiencia expandible.
- Pulir diagnostico.
- Mejorar recursos.
- Medir conversiones basicas.

### Fase 2 - Migracion a Next.js

Objetivo:

Convertir el sitio en una base moderna y mantenible.

Acciones:

- Migrar paginas actuales.
- Crear componentes reutilizables.
- Crear layout global.
- Crear menu/footer centralizados.
- Mantener diseno actual.
- Preparar rutas dinamicas.

### Fase 3 - Base de datos y panel admin

Permitir actualizacion de contenido sin tocar codigo.

Acciones:

- Conectar Supabase.

- Crear tablas para recursos.
- Crear tabla de leads.
- Crear panel `/admin`.
- Publicar/ocultar contenido.
- Subir imagenes.

## Decisiones pendientes

Estas decisiones no deben resolverse todas de inmediato, pero si conviene tenerlas presentes:

- Nombre final de la linea de merch corporativo.
- Si los servicios ampliados se mostraran en modal, pagina individual o ambas.
- Si los casos tendran URLs individuales para SEO.
- Herramienta inicial para formularios: Google Sheets, Make, Formspree, Supabase o backend propio.
- Herramienta de analitica: Vercel Analytics, Plausible, PostHog o Supabase.
- Momento exacto para migrar a Next.js.
- Si el panel admin sera construido a medida o con una herramienta tipo CMS.
- Estructura de precios para cursos/membresia.
- Politica de privacidad y consentimiento de datos.

## Primer backlog sugerido

Estas son tareas pequenas y ordenadas para empezar sin saltar directamente a una plataforma gigante:

1. Crear modal del Metodo ELEVAR.
2. Crear contenido completo de cada fase del metodo.
3. Crear modal o detalle ampliado para servicios.
4. Crear estructura de caso individual aunque sea estatica.
5. Conectar diagnostico a un destino real.
6. Agregar eventos basicos de conversion.
7. Definir primera version de politica de privacidad.
8. Preparar migracion a Next.js cuando el sitio estatico ya tenga el flujo validado.

## Criterios para decidir si algo entra ahora o despues

Una idea entra ahora si:

- Ayuda a captar prospectos.
- Mejora claridad comercial.
- Reduce friccion para el usuario.
- Es facil de mantener.
- Permite aprender algo medible.

Una idea queda para despues si:

- Requiere usuarios registrados.
- Requiere pagos.
- Requiere mucho contenido que aun no existe.
- Depende de un panel administrativo completo.
- Complica el sitio antes de validar conversion.

## Principio rector

No construir todo de golpe.

La prioridad debe ser avanzar por capas:

1. Sitio que convierte.
2. Contenido editable.
3. Leads y datos.
4. Recursos y autoridad.
5. Produccion de marca.
6. Membership.

La vision es grande, pero se puede construir de forma ordenada si cada fase entrega valor real antes de pasar a la siguiente.
